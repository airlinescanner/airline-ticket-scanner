# Workflow

## Главный принцип

Сначала архитектура → потом код.

Никогда не писать большие блоки кода без согласования.

Работа всегда идет этапами:

1. Предложил решение
2. Получил подтверждение
3. Реализовал
4. Проверили
5. Зафиксировали прогресс
6. Перешли дальше

# Memory System

## _SESSION_STATE.md

Файл `_SESSION_STATE.md` используется как память проекта.

После завершения этапа:
- записывать текущий статус
- фиксировать что сделано
- фиксировать что осталось

При старте новой сессии:
- сначала читать `_SESSION_STATE.md`
- предлагать продолжить работу с текущего состояния

# React / React Native правила

## Архитектура

- Только функциональные компоненты
- Использовать хуки
- Классовые компоненты не использовать
- Один компонент = одна ответственность
- Сложную бизнес-логику выносить в кастомные хуки
- Не мутировать state напрямую

---

# React Native правила

## UI и стили

- Использовать только `StyleSheet.create()`
- Не использовать inline styles
- Использовать Flexbox
- Избегать фиксированных размеров
- Учитывать `SafeAreaView`
- Для форм использовать `KeyboardAvoidingView`

---

## Производительность

- Для длинных списков использовать только:
  - FlatList
  - SectionList

Никогда:
- ScrollView.map()

---

## FlatList Optimization

Всегда использовать:
- keyExtractor
- initialNumToRender
- maxToRenderPerBatch
- windowSize

---

## Мемоизация

- Компоненты списков оборачивать в `React.memo`
- Колбеки оборачивать в `useCallback`
- Избегать анонимных функций в props

---

## Платформенная логика

При различиях платформ использовать:
- Platform.OS
- Platform.select()

---

# Next.js правила

## Архитектура

- Использовать только App Router
- Не использовать pages/
- Соблюдать структуру:
  - layout.tsx
  - page.tsx
  - loading.tsx
  - error.tsx

---

## Server First

- Компоненты серверные по умолчанию
- 'use client' только там, где реально нужна интерактивность

---

## Server Actions

Для мутации данных использовать:
- Server Actions

Не плодить лишние API routes без необходимости.

---

## Оптимизация

Всегда использовать:
- next/image
- next/font

Цель:
- минимизация Layout Shift
- улучшение Core Web Vitals

---

## SEO

Для страниц обязательно:
- metadata
- generateMetadata

---

## CSS

- Использовать CSS Modules
- Не использовать `!important`
- Соблюдать правильную специфичность селекторов

---

## Структура CSS файла

1. Базовые стили
2. Анимации
3. Media queries

Все `@media` размещать в конце файла.

---

# SEO для Reels / Shorts

## Семантика

Для каждого ролика:
- определять главный ключевой запрос
- использовать 3–5 LSI-ключей

Без переспама.

---

## Структура

### H1

- Первый хук видео
- Первая строка описания

Должны содержать главный ключ.

---

## H2

Подзаголовки описания использовать как H2.

---

## Accessibility

Всегда генерировать:
- Alt Text для видео

---

## Структура описания

Описание должно содержать:
- короткие абзацы
- списки
- CTA
- блок хэштегов

---

## Хэштеги

Использовать:
- 3–5 высокочастотных
- 3–5 нишевых

---
